远程线程注入–突破session0注入

1.什么是session0

session0 是一个专门的Windows 会话，其中完整显示/实例化Windows 和FireDaemon Pro服务的所有方面，包括交互式 GUI 组件（例如窗口、对话框、弹出窗口等）和其他方面（例如映射驱动器、环境变量等）与定期登录的 Windows 桌面会话隔离。这种隔离是有意设计的，并由操作系统强制执行。

而从Windows OS Kernel 6开始，采用session0隔离机制。例如在XP和Win7都登录了两个用户，但win7有三个会话（0，1，2），XP只有两个（0，1）。二者的系统进程和服务进程都在都在ID为0的session中运行，差别在于XP中第一个登录的用户进程的sessionID为0，Win7中，第一个登录的用户进程sessionID为1（非系统会话）。这样就隔离了服务和用户程序一起运行的风险，因为服务会使用提升后的权限（管理员）运行，用户应用程序的权限（非管理员）低，会使得恶意软件以某个服务为攻击目标，通过劫持服务，达到提权的目的。

2.为什么使用CreateRemoteThread向会话0服务进程注入失败

从WinXP开始，CreateRemoteThread内部实现算法采用挂起模式，即先创建线程，再使用“恢复运行”继续执行，若进程属于会话0，则不会恢复运行。

直接调用ZwCreateThreadEx时，此函数的第七个参数会发生改变，参数CreateSuspended为FALSE(0),而使用CreateRemoteThread去内部调用ZwCreateThreadEx,该参数变为TRUE(1)，此时线程挂起不再执行，这就是失败的原因。

补充：

通过调试kernel32.CreateRemoteThread（）跟踪到ntdll.ZwCreateThreadEx（），会发现它最终会通过SYSENTER指令进入内核，因此无法继续用户模式调试。

这是因为自从Windows Vista版本后（XP之前的版本不存在），新增了Kernelbase.CreateRemoteThreadEx()和ntdll.ZwCreateThreadEx()这两个API。

举个例子

WinXP的调用流程：kernel32.CreateRemoteThread（）–>ntdll.ZwCreateThreadEx()

Win7的调用流程：kernel32.CreateRemoteThread（）–>Kernelbase.CreateRemoteThreadEx() –>ntdll.ZwCreateThreadEx()
